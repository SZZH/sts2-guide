# StS2 发售周执行 Checklist

最后更新：2026-03-08

## 目标
- 复用现有 `app/news/[slug]` 流程，抢发售周高意图搜索流量。
- 优先保证上线速度和清晰内链，不在这一轮做新路由和新工具开发。

## 第一阶段：今天
为什么现在做：
- 发售后 24-48 小时是搜索意图最强的窗口，用户最先搜索的不是深度 build，而是“官方修了什么”“我为什么进不去”“能不能联机”“Deck 能不能跑”。
- 这批内容的共同特点是搜索急、点击意图强、对时效敏感。如果今天不上，后面再补，流量价值会明显下降。

这一阶段的目的：
- 抢占发售后最强的问题型流量和功能确认型流量。
- 让网站首页、launch hub、`/news` 三个入口都能把用户送到最急需的答案页。
- 尽快建立“这个站能解决发售周实际问题”的认知，而不是只像一个泛资料站。

希望达到的效果：
- 用户搜索补丁、报错、联机、Deck 相关问题时，站内已经有对应落地页承接。
- 首页和新闻页不再只展示泛内容，而是明显围绕发售周热点组织。
- 新发布页面之间形成互相导流，避免单页孤立。

## 第二阶段：未来 48-72 小时

## 第三阶段：未来 7 天

- 这时候再补机制细分页和 build 页，既不耽误前面抢时效流量，也能开始搭长期内容骨架。
- 对标 `sts2.wiki` 后，后续增长的关键不只是“多发文章”，而是把页面做成稳定模板（决策模块 + 内链模块 + 更新时间模块）。

## 单篇文章发布前检查
- 为什么要这样检查：
- 发售周文章不是写完就结束，真正目标是让它能被搜到、被点进来、被继续导流。
- 这些检查项的作用是确保每一篇都能同时满足搜索承接、点击转化和站内流转，而不是只变成一篇“发布了的文章”。

- 这一部分的目的：
- 保证每篇文章上线时都具备最基本的 SEO 完整性和内容可信度。
- 避免出现关键词不清、信息不确定、内链缺失这种会直接削弱流量效果的问题。

- 希望达到的效果：
- 每篇文章都能清楚回答一个主搜索问题。
- 每篇文章都能和站内其他关键页面形成连接。
- 每篇文章都能减少“写了但搜不到”或者“进来了但跳走”的问题。

- [ ] 标题包含主关键词
- [ ] 前两段直接回答用户搜索问题
- [ ] H1、title、description、canonical 与目标关键词一致
- [ ] 明确区分已确认信息、早期观察和未验证细节
- [ ] 至少包含一个 FAQ 或步骤式排查区块
- [ ] 至少包含 3 个站内链接
- [ ] 至少有 1 个链接回到 launch hub 或 `/news`
- [ ] 页面中有清晰可见的更新时间
- [ ] 文首附近有一段可被摘要提取的简短总结
- [ ] 页面中有作者信息（或统一编辑团队名），并与更新时间一起展示

- [ ] 图片来源可追溯（官方或可商用来源），并在页面末尾保留来源说明

## 图片来源策略（新增）
- 为什么现在加这一条：
- 竞品页面视觉更完整的关键之一是配图充分，但图片使用边界如果不先定清，会带来版权风险和后续下线风险。

- 这一部分的目的：
- 在提升页面可读性的同时，避免因图像版权问题影响收录、稳定性和品牌信誉。

- 希望达到的效果：
- 每个核心页面都有图且来源可追溯。
- 不依赖竞品站素材，避免侵权争议。

- [ ] 不直接搬运 `sts2.wiki` 的图片资源（含下载后再上传、或直接热链）
- [ ] 优先使用官方来源图片：Steam 官方商店素材、Mega Crit 官方新闻素材、官方媒体包
- [ ] 若使用社区图或第三方图库，先确认授权条款并保存证据
- [ ] 在文章底部加入简短来源说明（例如：Image source: Steam Store / Mega Crit）
- [ ] 为同一主题准备至少 1 张官方图和 1 张备用图，避免单链接失效

## 分发
- 为什么这样分发：
- 发售周最值钱的不是“广撒网”，而是把最能解决问题的页面送到最需要它的人面前。
- `known issues` 和 `co-op guide` 最容易在社区里触发真实需求和转发，优先级高于泛攻略页。

- 这一部分的目的：
- 用最少的分发动作放大最有传播性的页面。
- 把外部社区里的即时需求，引导到站内可承接的落地页。

- 希望达到的效果：
- 社区里有人问问题时，你有现成页面可以直接承接。
- 发出去的链接更容易被点，而不是发很多但没有反馈。

- [ ] 在问题导向的社区帖子里优先分发 `known issues`
- [ ] 在多人联机相关讨论里优先分发 `co-op guide`
- [ ] 发 1 条简短的 X 线程，总结发售周最重要的链接
- [ ] 发售后前 72 小时避免分发低紧急度页面

## 跟踪
- 为什么要跟踪：
- 发售周的判断不能靠感觉，必须用 GSC 和站内入口表现来判断哪些页面值得继续加码。
- 同样是上线 4 篇文章，真正有效的通常只有其中一部分，后续动作应该围绕结果加码，而不是平均投入。

- 这一部分的目的：
- 用最早期的数据判断页面标题、导语、承接词是否正确。
- 找出真正值得继续扩写和加入口的页面。

- 希望达到的效果：
- 你能明确知道哪篇文章在拿流量，哪篇文章只是挂着。
- 后续修改有依据，而不是凭主观判断改标题和内链。

- [ ] 检查发售周文章在 GSC 的 impressions、CTR 和 average position
- [ ] 跟踪首页和 `/news` 给各篇发售周文章带来的点击情况
- [ ] 对高展现低 CTR 页面迭代标题和导语

## 不要做
- 为什么现在不做：
- 发售周最大的成本不是“不会做”，而是“分心做太多”。新工具、新路由、重复意图页面，都会消耗最稀缺的窗口时间。

- 这一部分的目的：
- 把资源集中在最短时间内最可能见效的动作上。
- 避免因为结构扩张过早，拖慢第一批高价值页面上线。

- 希望达到的效果：
- 你的判断标准会更清楚：凡是不能直接提升本周搜索承接效率的，都暂时往后放。
- 这一轮执行会更聚焦，不会被“顺手做一下”拖偏。

- [ ] 第一轮发售周执行中不要开发新工具
- [ ] 第一批内容未上线前不要引入新路由结构
- [ ] 不要再发布一个与 starter character 意图重复的新页面
- [ ] 不要让泛 evergreen 页面挤占发售周紧急搜索意图
