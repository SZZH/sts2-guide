# Daily Runbook（按需执行版）

> 目标：避免机械跑模板，只做“当前有价值、且尚未完成”的动作。

## 1. 先判定，再执行
- 先读：`NOW.md`、`00-dashboard.md`、`01-daily-log.md`。
- 若本轮包含游戏动态巡检，再先读：`08-game-dynamics-intel.md`，先判定哪些已经关注过，哪些才是新增增量。
- 先输出三类结果：`已完成` / `进行中` / `未完成`（附证据）。
- 只执行“未完成差量动作”，不复述或重跑已完成动作。

## 2. 触发式动作清单（不是每次全跑）
- 四端数据巡检：仅当用户明确要求、距离上次结果较久、或出现异常信号时触发。
- Vercel 数据采集路由（强制）：后续统一通过 `Vercel MCP` 获取项目/部署/日志/分析数据；不再使用本地 `VERCEL_TOKEN` 脚本链路作为默认路径。
- 社媒评论处理：仅当存在待回复/超时积压时触发。
- 外链复核与补链：仅当有新内容或目标页确有缺口时触发。
- 游戏动态巡检：每日运维默认包含一次“近期游戏动态”检查，默认看近窗增量而不是只看当天新闻；目标是发现用户可能还没关注到、但会影响站点内容动作的变化。来源至少覆盖官方公告、Steam 新闻、SteamDB、官方社媒。
- 游戏动态去重：同一轮巡检默认先对照 `08-game-dynamics-intel.md`，优先补新增情报或更新已有条目状态，避免重复找同一批新闻。
- 版本更新巡检：若游戏动态巡检命中新补丁、热修、平衡调整、官方公告或社媒确认信息，立即升级为当日 P0。

## 3. 并行规则（按需）
- 当识别出 `>=3` 个互不依赖的未完成任务，优先拆 subagent 并行。
- 若不并行，先说明原因（依赖、写入冲突、限速、用户要求串行）。

## 4. 记录规则（最小必要）
- 用时间戳记录：`YYYY-MM-DD HH:mm`。
- 每次只回填本次新增信息，不要求把所有状态重新抄一遍。
- 有策略变化再写 `02-decisions.md`；无变化不强制新增决策记录。

## 5. 输出建议
- 推荐结构：`已完成证据 -> 当前缺口 -> 下一步差量动作`。
- 不要求固定 `DONE/P0/P1/SKIP` 模板；仅在需要排优先级时使用。
- 若本轮涉及数据监控/调研，追加三段：
  - `表现如何`：与上次或近 7 天相比的变化。
  - `为什么变化`：上涨/下滑的主要驱动因素与证据。
  - `建议怎么做`：下一步动作、优先级、预期影响。
- 若本轮包含游戏动态巡检，额外补一句：
  - `内容动作`：是否需要发新闻页、补内链、提交 IndexNow；若不需要，也要明确写“暂不动作”的理由。
  - `未关注增量`：这轮近期巡检里有哪些新变化是用户之前未明确提到、但值得关注的。
