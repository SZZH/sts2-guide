# Project Rules (sts2-guide)

## 部署触发规则（强制）
- 默认发布流程：代码合并/推送到 `main` 分支后，由 Vercel 自动触发生产部署。
- 非必要不手动执行 `vercel --prod`；仅在“紧急修复且用户明确要求立即手动发版”时才使用。

## Git 提交规则（强制）
- 后续所有 Git 提交信息（commit message）默认使用中文描述。
- 若无特别要求，提交标题与正文均使用中文，且应准确说明改动范围与目的。

## 数据调研规则（强制）
- 当我被要求做“数据监控/数据调研/增长分析/复盘”时，必须执行四端综合调研：`GSC + Vercel + Bing Webmaster Tools + Google Trends`。
- 当用户询问“今天/近期整体流量表现如何”时，必须先执行四端实时采集，再输出结论；禁止仅复用历史记录直接作答。
- 若任一平台当下无法实时采集，必须在结果中显式标记 `BLOCKED`（含原因、影响范围、补采时间），且不得把历史口径表述为实时结果。
- 调研深度要求：必须覆盖所有可能相关指标，不得只看单一点击或流量曲线。
  - GSC：点击、展现、CTR、平均排名、查询、页面、国家/地区、设备、搜索呈现、索引覆盖、结构化结果（Rich Results）状态。
  - Vercel（或站内分析）：Visitors、PV、Bounce Rate、Top Landing/Exit、页面趋势、异常波动时间段。
  - Bing Webmaster：Search Performance、Index Coverage、Sitemaps、URL Inspection、Crawl/Index 异常、IndexNow 接收状态。
  - Google Trends：核心词/品牌词/意图词趋势、区域差异、时间窗口变化、相关主题与相关查询。
- 输出格式要求：
  - 已完成动作必须显式标记（如 `DONE`）。
  - 未完成动作必须标记优先级（`P0/P1/P2`）和原因（如依赖平台状态、数据延迟、权限限制）。
  - 必须给出跨平台综合结论与后续执行清单，不能只给单平台结论。

## 数据时效判定规则（强制）
- 任何带有“今天/今日/现在/近期”语义的数据请求，默认视为“实时任务”，不得将跨天快照标记为“已完成”。
- “已完成”仅在以下条件同时满足时成立：`同自然日` + `同任务目标` + `无新增异常/无用户要求重跑`。
- 只要自然日发生变化（例如日志停留在 `2026-04-03`，当前日期为 `2026-04-07`），四端采集状态必须自动回退为 `TODO`，并重新采集后再输出结论。
- 历史数据只能用于“对比/趋势”，必须显式标注时间戳；禁止用历史口径替代当日实时口径。

## 今日状态写入闸门（强制）
- 未产生“当日真实执行证据”（如当日时间戳日志、当日快照文件、当日平台回执）前，禁止：
  - 将任务标记为 `DONE` 或 `IN_PROGRESS`
  - 更新 `00-dashboard.md` 中“今日状态”时间
  - 使用“今天已完成”措辞对外汇报
- 仅做历史核对时，必须使用“历史记录（截至 YYYY-MM-DD HH:mm）”表述；禁止映射为“今日已完成”。
- 若当日无执行，标准输出必须为：`DONE=无`、`IN_PROGRESS=无`、相关任务 `TODO`，并注明“待实时采集/待执行”。

## 验证闸门规则（强制）
- 任何 SEO 下一步动作（扩量发文、进入下一批页面、升级执行阶段）都必须先经过既定数据验证闸门。
- 若未达到当期闸门指标，即使我被要求“继续下一步”，也必须先明确提示“需先验证/补验证”，不得直接推进。
- 当前阶段（30 页验证批）闸门基线：
  - 收录率 `>= 60%`
  - 有曝光页面占比 `>= 40%`
  - 进入前 30 名页面数 `>= 6`
- 若未过线，默认只允许调整：词池、模板、内链；禁止盲目扩量。

## 结构改动冷静窗规则（强制）
- 当站点发生“结构性改动”（信息架构调整、主线入口替换、URL 收口、模板大改、批量页面重写）后，默认进入 `24-72h` 冷静观察窗。
- 在冷静窗内，禁止主动推荐新的上线改动（含 SEO 补位、承接改版、批量文案重写），除非用户明确要求立即执行。
- 冷静窗内允许动作仅限：回流观测、证据整理、归因分析、改动草案准备、已知阻塞排障。
- 若需要在冷静窗内建议小改，必须先满足：有明确异常证据 + 影响范围可界定 + 能说明不执行代价。
- 冷静窗结束后，才可基于回流证据决定是否推进下一轮实改。

## 社媒发布任务强制规则（Reddit / X / Steam）
- 当任务涉及 Reddit、X（Twitter）、Steam 社区的发帖、回帖、补链接、分发时，执行前必须先阅读：
  - `STS2_LAUNCH_EXECUTION_CHECKLIST.md` 中的 `模块六：社媒发布攻略（Reddit / X / Steam，必读）`
- 若未先阅读该章节，不得执行相关平台发布动作。

## 默认执行原则（跨平台）
- 默认优先“信息价值/讨论价值”，不要先做硬广导流。
- 每帖最多 1 个链接，且链接应指向单一落地页。
- 需按平台和社区规则做文案定制，禁止跨平台机械复用同文案。
- 遇到风控、过滤或移除时，先降推广感并改写，再考虑重发。

## 反模板执行原则（强制）
- 禁止把 runbook/checklist 当成“每次全量必跑”模板。
- 每轮仅执行与当前目标直接相关、且尚未完成的差量动作。
- 若某动作已在当日完成或明确进行中，且任务目标未变化，默认复用结果，不重复执行；跨天默认不复用“实时采集类”动作，必须重跑。

## 结果解读与建议规则（强制）
- 每次任务执行完成后，禁止只汇报“做了什么”。
- 必须同时给出：
  - 表现对比：至少与“上次记录”或“近 7 天基线”比较（上涨/下滑/持平）。
  - 原因分析：解释变化由哪些因素驱动（内容、渠道、抓取、时效、平台波动等）。
  - 下一步建议：给出可执行动作与优先级（如 P0/P1）。
- 收尾顺序固定：`当前情况分析 -> 原因判断（含置信度） -> 建议动作（含预期收益与不执行代价）`。
- 若数据不足以判断原因，必须明确“不足点 + 补采计划”，不得给空泛结论。

## 执行前历史核对规则（强制）
- 每次收到“执行任务”请求后，必须先读取历史执行记录，再决定是否执行；禁止先给待办清单。
- 最低必读文件：`docs/ops-center/01-daily-log.md`、`docs/ops-center/00-dashboard.md`，以及与当前任务直接相关的 `ops-logs/*` 记录。
- 必须先输出三类判定并附证据：`已完成` / `进行中` / `未完成`。
- 对 `已完成` 项，禁止再次提醒“去做一次”；仅允许汇报完成证据与可选下一步。
- 仅对 `未完成` 项执行动作；同日多次执行时，必须按时间戳追加更新（`YYYY-MM-DD HH:mm`），避免重复任务指令。
- 反机械复述闸门：凡当日已标记为 `DONE` 或明确 `IN_PROGRESS` 的任务，禁止再写成“接下来开始做 X”；必须改为“复用已有结果 + 补缺口动作”。
- 执行话术约束：行动声明必须指向“未完成差量”，不得复述全量标准流程（如再次宣称要先跑已完成的站点可用性/四端快照）。

## Subagent 并行拆分规则（强制）
- 当同一轮任务中出现“超过两个（>=3）完全不相干的独立任务”时，必须默认拆分为 subagent 并行执行。
- “独立任务”定义：目标不同、输入输出边界清晰、彼此无顺序依赖、不会共享同一关键写入点。
- 主线程职责：分配任务边界、汇总结果、做最终一致性检查与风险提示；不得把可并行任务无理由串行处理。
- 若判断“不适合拆分”（如强耦合共享上下文、存在关键路径阻塞），必须先说明原因再按单线程执行。
- 并行硬闸门：一旦判定存在 `>=3` 个独立任务，必须先输出 Subagent 分工表并立即分发，再执行主线程动作；不得先串行做子任务。
- 串行例外条件（仅允许以下情形）：共享同一关键写入点、强顺序依赖、外部平台速率限制/互斥锁、或明确用户要求串行。
- 若触发串行例外，必须在执行前写明：依赖关系、阻塞点、预计解除时间；否则视为流程违规。

## 社媒回帖格式规则（强制）
- 当任务涉及 Reddit / X / Steam 的“回复他人评论/回帖”时，默认采用“先引用对方原话，再给出回复”的结构。
- 引用应尽量抓取对方核心观点，避免断章取义；回复保持建设性、非对抗、可继续讨论。
- 如平台语法支持（如 Steam `[quote=用户名;评论ID]...[/quote]`），优先使用平台原生引用格式。

## GEO 执行准则（无侵入优先）
- GEO 相关优化默认不影响用户主阅读路径；不得为了抓取而新增大段干扰性文案块。
- 严禁使用仅给爬虫/模型可见、对用户隐藏的文本（包括 cloaking、隐藏 DOM、CSS 隐藏正文等）。
- 优先使用机器可读层提升 GEO：
  - `llms.txt`
  - JSON-LD（如 `Article` 的 `abstract`、`citation`、`about`、`inLanguage`）
  - 清晰且语义完整的现有正文与标题层级
- 内链优化优先采用“问题导向锚文本”，在不增加阅读负担的前提下提升查询意图匹配。
- 如 GEO 优化与阅读体验冲突，默认先保阅读体验，再选择结构化/内链的替代方案。
- 每次 GEO 改动后必须至少执行一次 `lint + build` 回归，确认无功能回归再进入下一步。

## 内容站持续更新规则（强制）
- 本站定位为持续更新内容站，禁止按一次性静态站方式执行。
- 当 Slay the Spire 2 出现版本更新、热修、平衡调整或官方公告时，必须及时抓取并发布新闻/解读内容。
- 默认每日按需多次巡检并留痕，来源至少覆盖：官方公告、Steam 新闻、SteamDB、官方社媒。
- 每次新闻发布后，必须同步执行：`IndexNow` 提交、Bing `URL Submission`、站内相关内链补位。
- 新闻页面必须与对应实体页（`/cards`、`/relics`、`/guides` 等）建立双向承接，避免孤页。

## 新页面发布规则（强制）
- 只要新增页面或新增可索引路由（如 `/builds`、`/tier-lists`、`/patches`），必须同步更新 `app/sitemap.ts`，确保新 URL 进入 `sitemap.xml`。
- 未完成 sitemap 更新前，不得将“新页面发布完成”标记为 `DONE`。
